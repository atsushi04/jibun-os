# フォルダカタログ（用途→フォルダのマッピング）

ヒアリング結果からフォルダ構成を組み立てるためのカタログ。
**必須フォルダは常に作成。条件付きフォルダはヒアリング結果に該当するものだけ**を提案する。番号は下表を基準にしつつ、ユーザーの構成に合わせて詰めてよい（原則1: 若い番号＝不変、大きい番号＝流動）。

## 必須（ヒアリング結果に関わらず作成）

| フォルダ / ファイル | 用途 | 備考 |
|:--|:--|:--|
| `CLAUDE.md` | AIの役割・フォルダの意味を書く中核コンテキスト | templates/CLAUDE.md.template から生成 |
| `DAILY.md` | 毎日の工程表・作業ログ | templates/DAILY.md.template |
| `00_context/` | 不変の文脈。`profile.md`（プロフィール）を含む | |
| `00_context/memories/` | AIの記憶。`MEMORY.md`（インデックス）を含む | templates/memories/MEMORY.md.template |
| `00_context/os-growth.md` | コンテキスト成熟度レベルの履歴 | os-growthスキルがLv1で初期化し、weekly/monthly-reviewの度に更新する |
| `01_strategy/` または `01_goals/` | 書き換え続ける目標・方針 | 仕事スコープあり→`01_strategy/`、生活のみ→`01_goals/`。スターターとして `goals.md`（ヒアリングで聞き取れた目標の骨子）を必ず置く。振り返りスキルの参照先が空にならないようにするため |
| `99_memo/` | 雑多なメモ・一時的な参考資料のインボックス | README.md（用途1行）を置く |

## 条件付き（ヒアリング結果で選択・代表例）

以下は**代表例**であり網羅ではない。Q2（活動プロファイリング）の回答に対しては、まずこの表への対応を試みる。対応する行がなければ「カスタムフォルダ」セクションのルールに従って新設する。

| フォルダ | 作成条件 | 用途 |
|:--|:--|:--|
| `02_projects/` | プロジェクト単位の仕事・活動がある | 案件・プロジェクト管理。`01_active/ 02_done/ 03_pending/` のステータスサブフォルダ構成（原則6） |
| `02_clients/`（projectsと排他か併用） | 顧客・顧問先を複数持つ（フリーランス・経営） | 顧客ごとの「文脈の島」（原則5）。`999_archive/` と横断一覧 `clients.md` を含む |
| `03_content/` | 記事執筆・SNS・ブログ等の発信をする | コンテンツ下書き・完成原稿。`tmp/`（一時置き場）付き |
| `04_finance/` | 事業の財務・請求管理をする | 財務メタデータ（CSV等）。原本PDFは外部ストレージ推奨と案内 |
| `05_insights/` | 顧客・現場から一次情報を蓄積したい（コンサル・企画系） | 現場の声・課題のDB |
| `06_learning/` | 学習・スキルアップの記録をつけたい | 学習ログ・カリキュラム |
| `08_life/` | 生活スコープを含む | 家計・旅行・健康など用途別サブフォルダ（例: `money/ trip/ health/ places/`）。ヒアリングで必要なものだけ作る |
| `tasks.md`（ルート） | タスク管理が「ローカルmd」の場合 | 未完了/完了の2セクション構成のタスクリスト |
| `WEEKLY.md` | 振り返り習慣に weekly を含む | 週次振り返りログ |
| `MONTHLY.md` | 振り返り習慣に monthly を含む | 月次振り返りログ |

## カタログにない用途への対応（カスタムフォルダ）

Q2の動的ヒアリングで、上記の代表例に当てはまらない回答（例:「研究記録を残したい」「患者さんとのやり取りを記録したい」）が出た場合は、無理に既存フォルダへ押し込めず、新しいフォルダを新設してよい。

- **番号**: 02〜08の空き番地から、その用途の不変度に応じて採番する（原則1: 若い番号＝不変、大きい番号＝流動）。例えば「顧客対応の記録」は`02`〜`03`帯、「学習記録」は`06`帯が自然
- **README.md必須**: 用途を1〜3行で書く（下記フォーマット参照）
- **CLAUDE.mdの構造表に反映**: 生成時にディレクトリ構造表へ1行追加する
- **新設は1回のセットアップで2個程度まで**: 最小の手数の原則（design-principles.md）を優先する。「他にも作れそうだけど、まずはこの2つで始めて、必要になったらいつでも足せる」と案内する
- **フォルダ名はユーザーの語彙を活かしてよい**: カタログの英語的な命名に縛られず、`03_research/`や`05_patients-insights/`のようにユーザーの言葉に近い名前にしてよい

## 生成しないもの（案内だけする）

- **機密PDF・契約書・領収書の原本**: Git管理外（Google Drive等の外部ストレージ）を推奨
- **APIキー・認証情報**: `.env` / `credentials/` に置きGit管理外とする旨をCLAUDE.mdの注意事項に含める

## 各フォルダのREADME.md

条件付きフォルダを作成するときは、直下に用途を1〜3行で書いたREADME.mdを置く。例:

```markdown
# 02_projects
進行中のプロジェクト・案件を管理する。1案件1フォルダ。
ステータスは 01_active / 02_done / 03_pending の置き場所で表す。
```
