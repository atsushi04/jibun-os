# ヒアリングガイド

ワークスペース構築前のヒアリングの質問設計と分岐ロジック。
**ユーザーが使っている言語で質問・応答すること**（このファイルは日本語だが、英語ユーザーには英語で聞く）。
AskUserQuestionツールが使える環境では積極的に使い、使えない環境では会話で1問ずつ聞く。

## 質問一覧（この順で聞く）

### Q1. 管理スコープ

「このワークスペースでAIに何を管理してほしい？」

| 選択肢 | 影響 |
|:--|:--|
| 仕事のみ | `01_strategy/`。生活系フォルダなし |
| 生活のみ | `01_goals/`。仕事系の条件付きフォルダは基本スキップ |
| 仕事＋生活 | `01_strategy/` ＋ `08_life/` |

### Q2. 活動プロファイリング（動的ヒアリング・最大3ラウンド）

Q1のスコープに応じて「仕事ブランチ」「生活ブランチ」のいずれか（仕事＋生活の場合は両方、続けて）を実施する。
**静的な4択では粒度が不揃いで、どれにも当てはまらないユーザーが離脱するリスクがある**ため、回答から次の質問をその場で生成する。ラウンドは**最大3回・早期終了OK**——具体的な構成を確信できたら2ラウンドで打ち切ってよい。

#### 質問生成の共通ルール

- 選択肢は2〜4個。AskUserQuestionなら「その他」は自動付与されるので明示不要、会話モードでは「当てはまらなければ自由に教えて」と一言添える
- 1問1トピック。次の質問の前に、前の回答をひと言で受け止めてから聞く（例:「なるほど、マネジメント寄りなんだね。じゃあ次に…」）
- 聞く目的は**フォルダに落とせる情報**に限定する。性格診断や雑談にしない
- どの選択肢にも当てはまらない回答は無理に既存カタログへ当てはめず、自由回答を要約して確認する
- 各ラウンドの回答は、後の設計案提示で「この回答からこのフォルダを提案してる」という根拠として使う（Step 2で明示する）

#### 仕事ブランチ（Q1で仕事を含む場合）

**R1（固定・入口質問）**:「今の働き方に一番近いのは？」

| 選択肢 | 次ラウンドの方向性 |
|:--|:--|
| 会社員（マネジメント寄り） | チーム運営・意思決定・育成の中身を掘る |
| 会社員（実務・専門職） | 成果物の種類・専門領域を掘る |
| フリーランス・個人事業 | 顧客との関わり方・案件の性質を掘る |
| 経営者 | 事業の意思決定・組織運営を掘る |
| その他（学生・研究・退職後の準備 等） | 自由記述をそのまま受けて、それに応じた質問を組み立てる |

**R2（動的生成・必須）**: R1の回答から生成。目的は「仕事の中身」を掴むこと——何を扱い（人・モノ・情報・お金）、何を成果物として出し（資料・コード・記事・意思決定）、誰と関わるか（チーム・顧客・一人）。

R1別の質問例（そのまま使ってもよいし、回答の言葉を借りて言い換えてもよい）:
- マネジメント寄り →「チーム運営で一番時間を使ってるのは？（例: 1on1・進捗管理・採用・育成）」
- 実務・専門職 →「仕事の成果物は主に何？（例: 資料・コード・設計図・記事）」
- フリーランス・個人事業 →「案件は何社くらい並行してる？顧客とのやり取りの記録は残したい？」
- 経営者 →「今一番時間を使ってる経営課題は？（例: 資金調達・採用・事業戦略・営業）」
- その他 →「その活動で、AIに記録・整理してほしいことは何？」

**R3（動的生成・条件付き）**: R2までの回答でフォルダ構成を確信できた場合はスキップしてよい。確信が持てない場合のみ、「日々の中で何を蓄積すると一番役に立ちそう？」（例: 顧客情報・学習記録・制作物・財務・現場の気づき）を聞く。

**マッピング**: 回答は`references/folder-catalog.md`の代表例への対応をまず試みる（例: 「チーム運営」→`02_projects/`、「顧客とのやり取りを残したい」→`02_clients/`、「資金調達」→`04_finance/`）。対応する代表例がなければ、`folder-catalog.md`の「カスタムフォルダ」ルールに従って新設する。

#### 生活ブランチ（Q1が生活のみ、または仕事＋生活の生活側。軽め）

**R1（固定・複数選択可）**:「生活の中で、AIに関わってほしい領域は？」

| 選択肢 | 追加候補フォルダ |
|:--|:--|
| 家計・資産 | `08_life/money/` |
| 健康・運動 | `08_life/health/` |
| 旅行・おでかけ | `08_life/trip/` |
| 学び・趣味 | `08_life/learning/` または独自フォルダ |
| その他 | 自由記述を受けて次のR2で深掘り |

**R2（動的生成）**: 選んだ領域それぞれについて「記録したい／計画を立てたい／AIに提案してほしい」のどれに近いか軽く確認する（複数領域を選んでいても、まとめて1問でよい）。R3は基本不要。当てはまらない領域が出た場合のみ、その領域について1問追加で深掘りする。

**マッピング**: `folder-catalog.md`の`08_life/`配下の代表サブフォルダにまず対応させ、なければカスタムサブフォルダを新設する。

### Q3. タスク管理の方法

「タスク管理はどうしてる？」

| 選択肢 | 生成スキルへの影響 |
|:--|:--|
| GitHub Issues と連携したい | daily-schedule / weekly-review / monthly-review が gh CLI でIssueを参照・集計する版になる。gh CLIのインストール・認証状態を確認し、なければ案内する |
| ローカルのMarkdownで管理したい | ルートに `tasks.md` を作成し、スキルは tasks.md を参照・更新する版になる |
| 別ツール（Notion等）を使っていて連携は不要 | スキルはタスクツールを参照せず、DAILY.mdの持ち越しと対話ベースで動く版になる |

「別ツール」を選んだ場合は、ツール名も一言聞く（例:「Notion」「Trello」）。CLAUDE.mdの説明文に使う。聞けなかった/答えがなければ空欄のまま埋めようとせず「外部ツール」という一般的な表現に留める（空括弧を残さない）。

### Q4. 振り返りの習慣

「振り返りはどこまで習慣にしたい？（あとから追加もできるよ）」

| 選択肢 | 生成スキル |
|:--|:--|
| 日次のみ | daily-schedule / daily-summary |
| 日次＋週次 | ＋ weekly-review（`WEEKLY.md` も作成） |
| 日次＋週次＋月次（推奨） | ＋ monthly-review（`MONTHLY.md` も作成） |

weekly以上を選んだ場合は、振り返り曜日も確認する（デフォルト: 土曜日）。特にこだわりがなければ聞かずに土曜日で進めてよい。
「日次のみ」「日次＋週次」を選んだ場合は、月次のフル診断（os-growthのfullモード）が自動では発生しない旨を後述のStep 4完了ガイドで案内するため、ここでは聞くだけでよい。

### Q5. AIの呼び方・トーン（任意・軽く）

「AIパートナーの呼び名や話し方の希望はある？（なければスキップでOK）」

- 回答があれば CLAUDE.md の「役割」セクションと `00_context/profile.md` に反映（例: 呼び名、敬語/フランク）
- なければデフォルト（丁寧すぎないフラットな日本語）で生成

### Q6. 生活リズム・目標・1日のタスク量（任意・軽く）

「平日/休日の時間の使い方や、意識してる大きな目標があれば教えて。あと1日に無理なくこなせるタスク数の目安はある？（特になければ1日3個で進めるよ）」

- 生活リズム・大きな目標の回答があれば `00_context/profile.md` と daily-schedule の `{{WORK_STYLE_NOTES}}` / `{{GOAL_REMINDER}}` に反映する
- タスク数の目安があれば daily-schedule の `{{DAILY_TASK_LIMIT}}` に反映する。回答がなければデフォルト値「3」を使う
- スキップされても深追いしない。空欄のまま進めてよい（os-growthの初回ヒアリングで後から拾える）

## ヒアリング時の心得

- **1メッセージに詰め込みすぎない**。AskUserQuestionなら最大4問まで、会話なら1問ずつ
- 回答から推測できることは聞き直さない（例: Q2のR1で「フリーランス」→ 財務管理はほぼ確実に必要なので、R2以降で聞くよりは確認を設計案の提示に含める形でよい）
- ユーザーの回答が選択肢に合わないときは、無理に当てはめず自由回答を要約して確認する
- Q2は動的ヒアリングのため、Q1〜Q6の中で唯一「ラウンド」という単位を持つ。Q2が終わったらQ3に進む
- ヒアリング完了後は**必ず設計案（フォルダ構成・生成スキル一覧・CLAUDE.md骨子）を提示して合意を得てから**生成に入る

## マイグレーションモード（既存ファイルがある場合）の追加ステップ

1. 既存のファイル・フォルダを一覧し、内容を軽く確認する（大きなファイルは先頭だけ）
2. ヒアリング結果のフォルダ構成に対し、**既存ファイル→新構成のマッピング表**を提示する
   - 「移動する」「そのまま残す」「判断つかないので確認」の3分類
3. ユーザーが合意した範囲だけ移動する。**削除は絶対にしない**
4. 既存の `.claude/skills/` や `CLAUDE.md` がある場合は上書きせず、追記・統合案を提示して確認を取る
